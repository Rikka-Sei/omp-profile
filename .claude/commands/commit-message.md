分析当前暂存的文件变更，生成符合 conventional commits 规范的提交消息，执行实际提交，等待用户确认。

## 执行步骤

1. 检查暂存的文件变更：
`git diff --staged --name-status`

2. 查看详细变更内容：
`git diff --staged`

3. 分析变更并生成提交消息，格式为：
   ```
   <type>: <简短描述>

   - <变更详情1>
   - <变更详情2>
   ```

4. 提交类型选择：
   - `feat:` - 添加新功能
   - `fix:` - 修复 bug 或问题
   - `refactor:` - 重构代码
   - `docs:` - 文档更新
   - `test:` - 添加或修改测试
   - `chore:` - 构建过程或辅助工具的变动

5. 询问用户是否确认提交：
   - 使用 question 工具询问用户是否确认执行提交
   - 提供"确认提交（无GPG KEY）"选项
   - 提供"确认提交（默认GPG KEY）"选项
   - 提供"确认提交（预置GPG KEY: 3927D7）"选项
   - 提供"取消"选项

   1. 如果用户选择"确认提交（无GPG KEY）"，执行提交命令：
      ```bash
      git commit -m "<生成的提交消息>"
      ```

   2. 如果用户选择"确认提交（默认GPG KEY）"，执行签名提交命令：
      ```bash
      git commit -S -m "<生成的提交消息>"
      ```

   3. 如果用户选择"确认提交（预置GPG KEY: 3927D7F5365B0203）"，执行签名提交命令：
      ```bash
      git -c user.signingkey=3927D7F5365B0203! commit -S -m "<生成的提交消息>"
      ```

   4. 如果用户选择"取消"，则不执行任何 git 命令

## 约束规则

- 使用中文描述变更内容
- 遵循 conventional commits 规范
- 提交消息简洁明了
- **禁止 AI 标识**：提交消息中不得包含任何表明是 AI 助手完成的标记

## 使用场景

当你想要：
- 自动生成提交消息并等待确认
- 保持提交消息的一致性和规范性
- 使用 GPG 签名提交代码
